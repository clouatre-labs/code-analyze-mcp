# v13: OpenFAST AeroDyn Integration Audit Benchmark

## Overview

v13 measures the impact of MCP tools versus native tools on Fortran scientific HPC code analysis. It
uses the same 2x2 factorial design as v12 (model x tool_set) with 4 conditions, scored across 3 rubric
dimensions, and analyzed for tool-set effects on task performance.

The target repository is OpenFAST (`OpenFAST/openfast`), NREL's wind turbine and wind-farm simulation
suite. It comprises 344 Fortran source files across 20+ physics modules, with a shared utility core
(`modules/nwtc-library/`) used by every physics module. This structure is deliberately harder for native
file-reading tools than Django was: subroutine names like `CalcOutput` and `UpdateStates` appear in
every module, so grep returns unscoped noise. `analyze_symbol` with a directory path resolves the
correct module in one call.

## Background

v12 established MCP advantage on Python (Django): 47% fewer tokens and 2x faster on Sonnet 4.6, 16%
fewer tokens and 40% faster on Haiku 4.5. v13 tests whether the same advantage holds on Fortran
scientific HPC code, where:

- Subroutine names are not namespaced (same name in every module)
- Call chains span multiple physics modules and the shared NWTC library
- Type definitions live in separate `_Types.f90` files generated by a preprocessor
- The codebase is 344 files; grep-based exploration is proportionally more expensive

### Why this codebase highlights code-analyze-mcp

1. **Name collision problem.** Every physics module exports `CalcOutput` and `UpdateStates`. A grep for
   `subroutine CalcOutput` returns hits in all 20+ modules. `analyze_symbol` with a scoped path returns
   only the AeroDyn definition and its callees.

2. **Type explosion.** Each module has a corresponding `_Types.f90` file with dozens of derived type
   definitions. Tracing which NWTC types AeroDyn uses requires opening 5-10 files with native tools;
   `analyze_file` on `AeroDyn_Types.f90` returns all type declarations and their line numbers in one
   structured response.

3. **Call chain depth.** AeroDyn's `AD_CalcOutput` calls BEMT, which calls NWTC math routines.
   Native tools require iterative grep-read-grep loops. `analyze_symbol` with `follow_depth=2` collapses
   this to a single call.

4. **Signal-to-noise ratio.** A 344-file Fortran repo contains more grep noise per useful hit than
   the Django auth subtree did. The MCP advantage on precision is proportionally larger.

## Task

**Title:** OpenFAST AeroDyn Integration Point Audit

**Context:**
You are onboarding to the OpenFAST codebase to extend the AeroDyn module with a new blade-load
correction factor. OpenFAST is a physics-based wind turbine simulation framework. AeroDyn is the
aerodynamics module responsible for computing blade loads. It follows the FAST modular framework
interface: every physics module exposes `Init`, `CalcOutput`, `UpdateStates`, and `End` subroutines
with module-prefixed names (e.g., `AD_CalcOutput`).

**Your task:**
1. Identify the exact Fortran files and subroutine names (with line numbers) that define AeroDyn's
   top-level `CalcOutput` and `UpdateStates` routines (look for `AD_CalcOutput` and `AD_UpdateStates`
   or similar module-prefixed names in `modules/aerodyn/src/AeroDyn.f90`).
2. Trace which NWTC library routines (from `modules/nwtc-library/src/`) are called within
   `AD_CalcOutput` -- up to 2 call levels deep. For each routine, name it and the file it is defined in.
3. Identify which derived types (Fortran `TYPE` definitions) from `modules/nwtc-library/src/` are
   used by AeroDyn. For each type, name the file and approximate line number where it is declared.
4. Produce an integration map: which files and line ranges must change to inject a new scalar
   correction-factor parameter (`BladeLoadCorrFactor`) through the call stack -- from the glue-code
   entry point in `modules/openfast-library/src/FAST_Subs.f90` down to the blade-element calculation
   inside AeroDyn.

**Output schema (JSON):**
```json
{
  "run_id": "<condition>-<pilot|scored>-<N>",
  "condition": "A|B|C|D",
  "aerodyn_entry_points": [
    {"subroutine": "AD_CalcOutput", "file": "path/relative/to/openfast/root", "line": 0}
  ],
  "nwtc_callees": [
    {"routine": "name", "file": "path/relative/to/openfast/root", "call_depth": 1}
  ],
  "nwtc_types_used": [
    {"type_name": "TypeName", "declared_in": "path/relative/to/openfast/root", "approx_line": 0}
  ],
  "integration_map": [
    {"file": "path/relative/to/openfast/root", "line_range": "start-end", "change": "description"}
  ],
  "tool_calls_total": 0
}
```

## Execution

All runs are executed via `scripts/bench-v13-run.sh`, called from goose (the orchestrator). Each run
is a single non-interactive `claude` invocation (Claude Code CLI, `claude -p`) with tool isolation
enforced at the CLI level. The goose session invokes the script; it does not participate in the
analysis task itself.

**Runner:** Claude Code CLI (`claude -p`)
**Orchestrator:** goose (calls `bench-v13-run.sh` for each run in sequence; reads telemetry and
report JSON after each run completes)
**Tool isolation:**
- MCP conditions (A, C): `--mcp-config docs/benchmarks/v13/mcp-code-analyze-only.json --strict-mcp-config --allowedTools "mcp__code-analyze__*"`
- Native conditions (B, D): `--mcp-config <empty> --strict-mcp-config --allowedTools "Bash,Glob,Grep,Read,Write,ToolSearch"`

`--strict-mcp-config` prevents Claude Code from loading any MCP servers not listed in the config
file. The empty config for native conditions ensures the `code-analyze-mcp` server is not available
even if the user's global Claude Code config has it enabled.

**Structured output:** Each run uses `--json-schema` to enforce the output shape. The runner extracts
the `structured_output` field from the result envelope and writes it to `results/runs/<RUN_ID>-report.json`.
Telemetry (tokens, cost, turns, wall time) is written to `results/runs/<RUN_ID>-telemetry.json`.

**Commit pinning:** The runner checks out `$OPENFAST_COMMIT` before every run. Scored runs abort if
the checkout fails; pilot runs warn and continue.

## Conditions

### MCP conditions (A, C)

**Allowed tools:**
- `analyze_directory`
- `analyze_file`
- `analyze_symbol`
- `analyze_module`

**Forbidden:**
- Glob, Grep, Read, Bash, and any other tools

**Recommended call sequence:**
1. `analyze_directory(path="<repo>/modules/aerodyn/src", max_depth=2, summary=true)` -- orient (1 call)
2. `analyze_file` on `AeroDyn.f90` -- find `AD_CalcOutput` and `AD_UpdateStates`
3. `analyze_symbol(path="<repo>/modules/aerodyn/src", symbol="AD_CalcOutput", follow_depth=2)` -- trace
4. `analyze_directory(path="<repo>/modules/nwtc-library/src", max_depth=1, summary=true)` -- NWTC types
5. `analyze_file` on 1-2 NWTC type files identified above
6. `analyze_file` on `modules/openfast-library/src/FAST_Subs.f90` -- glue code entry point

### Native conditions (B, D)

**Allowed tools:**
- `Glob` (file pattern matching)
- `Grep` (content search)
- `Read` (file content)
- `Bash` (shell execution)

**Forbidden:**
- `analyze_directory`, `analyze_file`, `analyze_symbol`, `analyze_module`, and any other tools

## Rubric

### Dimensions and Scoring

Three dimensions, each scored 0-3 (max total = 9). No tool-efficiency dimension in the rubric score
(tool call counts are recorded as descriptive metadata only).

#### Dimension 1: Structural Accuracy (0-3)

Ability to correctly identify the Fortran file and subroutine structure of AeroDyn's FAST-interface
entry points.

- **0:** No identification of `AD_CalcOutput` or `AD_UpdateStates`; incorrect file paths; confuses
  AeroDyn with another module
- **1:** Identifies `AeroDyn.f90` as the key file; names at least one entry-point subroutine
  (`AD_CalcOutput` or `AD_UpdateStates`) but without a correct line number; may conflate with
  sub-module routines (BEMT, FVW)
- **2:** Correctly names both `AD_CalcOutput` and `AD_UpdateStates` with correct file path; line
  numbers within ±20 lines of actual; correctly distinguishes the module-level entry points from
  internal solvers
- **3:** Correctly names both entry points with exact file path and line number within ±5 lines;
  identifies the `_Types.f90` companion file pattern; notes that `AeroDyn_Types.f90` is
  auto-generated and contains type definitions

**Calibration (commit 2895884, verified by pilots A-pilot-1 and A-pilot-2):**
- `AD_CalcOutput`: `modules/aerodyn/src/AeroDyn.f90` line 2148
- `AD_UpdateStates`: `modules/aerodyn/src/AeroDyn.f90` line 1830
- Companion type file: `modules/aerodyn/src/AeroDyn_Types.f90` (registry-generated)
- Intermediate dispatcher: `RotCalcOutput` in `AeroDyn.f90` ~line 2200 (called by `AD_CalcOutput`)

#### Dimension 2: Call Chain Tracing (0-3)

Ability to trace the call chain from `AD_CalcOutput` through AeroDyn's internal routines and into
the NWTC library, and to identify NWTC types used by AeroDyn.

Score on correctness of the call chain regardless of method used to discover it.

- **0:** No identification of any intermediate routines or NWTC library routines; no mention of
  `RotCalcOutput`, `BEMT_CalcOutput`, or `modules/nwtc-library/`
- **1:** Identifies that NWTC library is used or names `BEMT_CalcOutput` as an intermediate; fewer
  than 2 specific routines named with file-level evidence
- **2:** Names `RotCalcOutput` and `BEMT_CalcOutput` with correct file paths; names 1-2 NWTC library
  routines (e.g., `SetErrStat`, `MeshCreate`); identifies 2-3 NWTC types with source files; minor
  gaps in depth coverage
- **3:** Names `RotCalcOutput` + `BEMT_CalcOutput` + at least 2 NWTC library routines
  (`SetErrStat`, and at least one of `MeshCreate`/`MeshCommit`/`MeshCopy`/`MeshMapCreate`/
  `Transfer_Line2_to_Point`) with correct file paths; identifies 3+ NWTC types with file evidence;
  traces at least one depth-2 path (AD_CalcOutput -> RotCalcOutput -> NWTC routine)

**Calibration (commit 2895884, verified by pilots A-pilot-1 and A-pilot-2):**
- `AD_CalcOutput` calls `RotCalcOutput` (internal, `AeroDyn.f90` ~line 2200)
- `RotCalcOutput` calls `BEMT_CalcOutput` (`modules/aerodyn/src/BEMT.f90`)
- NWTC library routines reached: `SetErrStat` (`NWTC_Base.f90`), `WrScr` (`NWTC_IO.f90`),
  `MeshCreate`/`MeshCopy`/`MeshCommit` (`ModMesh.f90`), `MeshMapCreate`/`Transfer_Line2_to_Point`/
  `Transfer_Point_to_Point`/`Lump_Line2_to_Point` (`ModMesh_Mapping.f90`)
- NWTC types used by AeroDyn: `MeshType` (`ModMesh_Types.f90`), `MeshMapType`
  (`ModMesh_Mapping.f90`), `ProgDesc`/`FileInfoType`/`OutParmType`/`ModVarType`/`ModVarsType`/
  `ModJacType` (`NWTC_Library_Types.f90`)

#### Dimension 3: Integration Map Quality (0-3)

Quality of the proposed change map for injecting `BladeLoadCorrFactor` through the call stack.

- **0:** No integration map proposed; no files cited; no mention of parameter propagation
- **1:** Mentions `AeroDyn.f90` as the only change point; does not trace the parameter from glue code
  to blade element; no line numbers; does not address the `_Types.f90` type-generation pattern
- **2:** Identifies 3-4 files that must change (e.g., `FAST_Subs.f90`, `AeroDyn.f90`,
  `AeroDyn_Types.f90`, `BEMT.f90`); provides approximate line ranges for at least 2 files; notes
  that types flow through the FAST modular interface (input/output/parameter structures); minor
  gaps in tracing the full stack
- **3:** Complete integration map covering all 4+ layers:
  - Glue code: `modules/openfast-library/src/FAST_Subs.f90` -- where to pass the parameter into
    `AD_CalcOutput` call
  - AeroDyn interface: `modules/aerodyn/src/AeroDyn.f90` (`AD_CalcOutput` signature change)
  - AeroDyn parameter type: `modules/aerodyn/src/AeroDyn_Types.f90` (or parameter struct)
  - Sub-solver: `modules/aerodyn/src/BEMT.f90` or equivalent blade-element routine
  - Provides line ranges for at least 3 of the 4 files; explains the FAST type-passing convention
    (parameters flow through the `p` parameter struct, not as naked subroutine arguments)

**Calibration notes (commit 2895884, verified by pilots A-pilot-1 and A-pilot-2):**
- The FAST modular framework passes parameters via the `p` (parameter) struct, not as naked subroutine
  arguments; a correct integration map notes this convention
- `AeroDyn_Types.f90` is auto-generated from a Registry file (`AeroDyn_Registry.txt`); a high-scoring
  response notes this and directs the change to the Registry file rather than the generated file
- Ground-truth touchpoints: `FAST_Subs.f90:2676-3440` (input reading), `FAST_Subs.f90:107-1563`
  (init forwarding), `AeroDyn.f90:1447-1584` (SetParameters), `AeroDyn.f90:2228-2338`
  (RotCalcOutput), `AeroDyn.f90:4068-4148` (SetOutputsFromBEMT), `BEMT.f90:1271-1416`
  (blade-element level, optional deeper injection)
### Scoring process

1. **Pilot protocol (stop criteria):**
   - No AeroDyn file paths cited anywhere in output -> Score 0 on Dimension 1, proceed to scoring
   - JSON output unparseable or missing required fields -> Record failure type, mark run invalid
   - All 0 on integration_map (no files cited) -> Stop and retry task description or model
   - All 3 on all dimensions -> Rubric may be too lenient; flag for re-calibration

2. **Scoring:** Two independent reviewers score each run; report median score and any disagreements.

## Tool Call Analysis

Record `tool_calls_total` in output JSON. This is **descriptive metadata only**, not a rubric score.
Used to observe differences between conditions (e.g., do MCP tools require fewer calls?).

For the Fortran task, the expected pattern (calibrated from the v13 pilot) is:
- MCP conditions: 20-35 tool calls (directory survey + paginated file/symbol lookups)
- Native conditions: 30-60 tool calls (iterative grep-read cycles to resolve name collisions)

## Analysis

- **Primary analysis:** Rank-biserial correlation `r` between condition (MCP vs native) and total score
- **Secondary:** Descriptive statistics by condition (median, IQR, tool call counts, token counts)
- **No p-values:** n=8 is too small for frequentist inference; report effect sizes and CI only
- **Exploratory:** Separate analysis by model (Sonnet vs Haiku) to detect interactions
- **Cross-benchmark comparison:** Compare MCP savings ratio against v12 (Django) to assess whether
  Fortran HPC code amplifies or attenuates the MCP advantage
